## 金字塔总结

**核心结论（一句话）**
在 Delivery 域引入 **LMO 作为域内基础订单元数据**，通过「优先查 LMO、兜底 FO」解除对 FO 的强依赖，提升全链路**稳定性与性能**，并支撑**十亿级单量**与**万级 DB QPS**。

**三大支撑（论据自上而下展开）**

| 层级              | 要点                                                                                                                                                                                                                                 | 对应后文               |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------- |
| **1. 问题** | FO 承载十亿级订单数据（**ID 日常约 1000 万单、大促约 2000 万单**；**全市场日常约 2500 万单、大促约 5000 万单**），各域直连查询导致调用集中（大促读 FO 接口峰值数千、部分场景到 FO DB 约 6 万 QPS），单点故障影响全链路。 | 一～二、四（封底估算） |

| **3. 结果** | 域内优先 LMO、兜底 FO，降低对 FO 依赖与调用量；支撑多市场半年约 10 亿+ 单量、ID 大促 DB 读峰值约 2.5 万 QPS（两年后约 5 万）；LMO 查询 RT&lt; 100ms、事件处理 RT &lt; 200ms、失败率 &lt; 0.1%；具备完整灰度与数据一致性验证能力。    | 四、七、八             |

## 一、背景

在 SPX 业务架构中，**FO（Fulfillment Order）** 作为订单元数据的管理方，负责为其他模块提供订单数据的查询能力。各作业域（如 Delivery、Pickup 等）均依赖 FO 获取订单基础信息与状态。

**Delivery 域内的缺口（与汇报材料一致）**：原 **Delivery Order** 的定义主要覆盖**骑手派送作业**相关阶段；域内还有 Route、AT、集单/运营等能力，但缺少能覆盖**全作业周期**的**基础订单模型**，导致 Delivery 域与 **Order Center / FO** 强耦合、难以独立演进。

---

## 二、冲突

- **稳定性风险**：整体架构高度依赖 FO 的稳定性，FO 一旦故障会影响**全部作业模块**的主流程。
- **负载集中**：全部作业模块都直接查询 FO，对 FO 形成较高负载，不利于扩展与容灾。

---

## 三、解决方案

为了提升 SPX 服务的**稳定性**和**性能**，最终决定在**各作业域内保存一份本域的数据模型**，解除对 FO 的强依赖。
在 Delivery 域，该数据模型即 **LMO（Last Mile Order）**。

### 3.0 功能框架（与 llmo.pptx「LMO - Functional Framework」对齐）

LMO 在 Delivery 域内承担的核心能力可归纳为：

| 能力 | 说明 |
|------|------|
| **事件消息处理** | 支持 **Plan 事件**与 **Delivery 域内作业事件** 的处理（异步、Kafka）。 |
| **生命周期管理** | 覆盖 Delivery 域内订单的**全生命周期**（状态机 + 数据校验）。 |
| **订单查询** | 支撑域内**全作业周期**的订单查询（基础查询与复杂查询场景）。 |
| **灰度管理** | 读对比灰度与切流灰度（多维度匹配、万分之一精度等）。 |
| **对账与修数** | 多阶段对账、本地协程池异步对比、DataFix 工具。 |

与调度中心、Delivery Order Center、Event Center 等的关系：Dispatch 下发 Plan；LMO 消费 Plan/域内事件；查询侧优先 LMO，**兜底再查 FO**（汇报图中的 fallback query）。

### 3.1 带来的好处

1. **减轻 FO 压力、降低系统风险**：各域优先使用域内数据，减少对 FO 的调用量与依赖面。
2. **提高服务稳定性**：**优先查 LMO**，仅在数据不一致时**兜底查询 FO**，单点故障影响面可控。

### 3.2 带来的问题：数据一致性

由于在域内冗余了数据，必然存在**数据不一致**的可能，需要在模型与流程上显式应对。

#### LMO 领域模型数据来源（与 llmo.pptx「LMO - Domain Model」三分类）

| 分类 | 来源与更新方式 |
|------|----------------|
| **基础数据（Basic data）** | 来自订单侧；通过 **Plan** 增量更新。 |
| **Plan 数据（Plan data）** | 来自 **Dispatch（调度中心）** 下发的 Plan。 |
| **Delivery 域内作业数据（Delivery field data）** | 来自 Delivery 域内**作业过程**产生的信息（骑手、站点、任务等）。 |

（与下文「核心基础数据 / 收件发件人」等字段级清单互补，字段明细仍以 Confluence 与 spreadsheet 为准。）

#### FO 数据模型（LMO 需同步的维度）

- **核心基础数据**：订单标识、状态、业务类型等。
- **收件人、发件人信息**：地址、联系方式等。
- 其他 Delivery 域内查询 FO 时依赖的字段（详见 Confluence 与需求 spreadsheet）。

#### FO 数据的更新场景（对应 LMO 的更新来源）

| 场景               | 说明                                                                                                                                               |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| **FO 更新**  | FO 侧发起的更新，通过**异步通知**（现有 **Plan 机制**）通知各作业域，各域做**增量更新**。                                        |
| **内部更新** | LM（Last Mile）内部作业过程产生的写入（如状态变更、骑手信息等）：**对外**通过 Plan 回写/同步，**对内**通过**异步任务**更新 LMO。 |

### 4.2 读 QPS

- **77 大促读 FO 最大 QPS（CAT 监控）**：ID 2000、PH/TH/MY 各 400、VN 1200、TW 10、SG 100、BR 200。
- **接口 QPS 峰值预估**：以 CAT 1 小时平均值为基 ×1.2，再预留 30% buffer。示例（77 大促）：ID 接口 3120、DB 读 24960（LMO 共 8 张表，单次接口平均查 8 张表）；两年后按单量翻倍，ID DB 读约 **49920**。

### 4.3 写 QPS

- **77 大促写 QPS 构成**：CreateFleetOrderTrack + Delivery Plan 等。各市场写 QPS 总和示例：ID 750、VN 210、BR 130 等。
- **接口写 QPS 峰值**：总和 ×1.2（峰值系数）×1.3（30% buffer）。示例：ID 接口写约 1170，DB 写约 **4680**（单次平均写 4 张表）；两年后 ID DB 写约 **9360**。

### 4.4 预估 DB 容量（结论）

- **数据量**：按市场与保留周期（如半年）汇总，单表行数、磁盘与 IOPS 依分库分表（见下节）反推。
- **读**：大促/高峰 DB 读 QPS 以「接口 QPS × 8」计（8 张表）；ID 当前约 2.5 万级，两年后约 5 万级。
- **写**：DB 写 QPS 以「接口 QPS × 4」计（平均写 4 张表）；ID 当前约 4.7k 级，两年后约 9.4k 级。

### 5.1 DB 设计

#### 分库规则

- 按 **shipment_id** 做哈希，将数据分散到多个库，降低单库压力、便于水平扩展。
- 方案：**10 个库**（参考 FO 分库分表与峰值 QPS，如 ID 当前 DB 读约 2.5 万、两年后约 5 万）。

#### 分表规则

- 同一库内按分片键哈希分表，控制单表数据量与查询 RT。
- **方案 A（Confluence / lmo_db.xlsx）**：按 **shipment_id**，**10 库 × 每库 100 表**（合计千级子表规模，与「10 库×100 表」表述一致）。
- **方案 B（llmo.pptx「LMO - Storage Design」）**：独立库 `shopee_delivery_lmo_xx_db` 等，**10 个 DB、共 1000 张表**（`xx_tab_000`～`xx_tab_999` 示意），分表键表述为 **order_id**；ID 市场半年数据量约 **10 亿级**，DB 读峰值约 **50k**、写峰值约 **9k**（与封底估算中「两年后约 5 万读 / 9.4k 写」同量级，具体以线上与评审结论为准）。
- **LMO 逻辑表数量**：共 **8 张表**，单次接口查询平均访问 8 张表，写入平均涉及 4 张表（见 `lmo_db.xlsx` 备注）。

*（详细表结构、字段说明见 Confluence 与 [LMO 表结构设计](https://confluence.shopee.io/x/wNysbQ)、相关 Spreadsheet；**分表键以最终表设计为准**。）*

### 5.2 Kafka 资源（来自 lmo_db.xlsx「kafka规格评估」）

- **用途**：作为事件总线，处理 Delivery 域内订单相关事件（Plan + 域内异步任务）。
- **Topic 命名**：按市场拆分，如 `spx-delivery-lmo-event-{id|ph|vn|br|th|my|sg|tw}-live`。
- **规格示例**（单市场）：
  - **高流量市场**（ID/PH/VN/BR）：峰值约 2400 QPS，消息均约 1KB，TPS 约 **2.4 MB/s**，保留 2 天存储约 **406 GB**。
  - **其他市场**（TH/MY/SG/TW）：峰值约 1200 QPS，TPS 约 **1.2 MB/s**，保留 2 天存储约 **203 GB**。
- **属性**：核心 Topic、核心链路、不可延迟消费。

## 六、核心业务流程

### 6.1 LMO 数据的更新场景

与「FO 数据更新场景」一致，LMO 的更新来源分为两类：

| 来源                 | 机制                                                                                                                                                       |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **FO/Plan 侧** | 通过现有**Plan 机制**（Create/Update/Complete/End/Cancel）异步通知 Delivery 域，LMO 消费 Plan 事件做**增量更新**（创建、状态扭转、字段更新）。 |
| **域内作业**   | LM 内部作业产生的写入：**对外**通过 Plan 回写；**对内**通过 **异步任务**（如 Kafka 消费）更新 LMO，保证与 FO 最终一致。                  |

多 Plan 类型（Delivery Plan、Collection Pool Plan、Return Locker Order Plan、DataFix Plan、Interception 等）的差异化逻辑、乱序与幂等（同 order 同 partition、最后消费时间过滤等）见 Confluence 事件处理与状态机章节。

**与 llmo.pptx「LMO - Core State Machine」**：状态扭转由 **Plan 触发**与 **Delivery 作业触发**两类来源共同驱动，需在状态机层统一校验。

**与 llmo.pptx「LMO - Event Message Handling」**：LMO 数据**创建、更新、状态扭转**均事件驱动；**Kafka 异步**处理；ID 市场事件峰值约 **2400 QPS**（与 Kafka 规格评估一致）。事件体字段：**OrderIds、MsgId、EventCode、EventTime、EventContent**（JSON，随事件类型变化）。

**与 llmo.pptx「LMO - Data Consistency」**：消费侧判断消息 **Operation time** 是否晚于 LMO 最新操作时间；满足状态机才更新；否则忽略或报错等待重试。要点：**同订单消息进同一 partition**、**同订单按操作时间顺序处理**、结合 **Plan 号 + 操作时间 + 状态机** 做幂等、**限时重试** 消化乱序。

### 6.2 LMO 的数据查询

- 域内**优先查 LMO**；查不到或需兜底时再查 FO。
- 打 AT/RAT 等场景由「基于 Plan 校验」改为**基于 LMO 校验**，兜底逻辑收口在 LMO 侧。
- 查询 RT 目标：**&lt; 100ms**；失败率 **&lt; 0.1%**。

### 6.3 数据对账

- **LMO 与 FO/作业信息一致性**：定时任务对非终态 LMO 做状态与关键字段校验（含正逆向），发现不一致则告警或触发修数。
- **Plan 数据一致性**：通过读对比（双写/双读阶段）校验 Plan 下发的 FO 字段与 LMO 落库数据是否一致。
- **修数工具**：支持按 order id 批量创建/更新 LMO，用于灰度与线上数据修复。

**与 llmo.pptx「LMO - Reconciliation Strategy」对齐**：全字段、多阶段**准实时对账**；**本地协程池**异步对比；可按场景配置**忽略部分字段**；对账准确率达标（汇报材料为 **99.99%**）后再启动切换。阶段示意：**Phase 1** 创建 LMO 数据；**Phase 2** 更新 LMO + Plan 数据对比 + Delivery 作业数据对比；**Phase 3** 使用 LMO 为主 + 多场景对比（含查 FO、组装场景等）。

### 6.4 灰度过程

- **读对比**：先双读 LMO 与 FO，对比结果，**连续 2 天 100% 一致**后再切流到仅读 LMO（与 Confluence 一致；与 ppt 的 **99.99%** 阈值可同时作为「达标线」表述，以实际发布门禁为准）。
- **灰度策略**：按 **shipment_id**（或容器号等）哈希，万分之一精度；支持多维度匹配与白名单应急。

**与 llmo.pptx「LMO - Grey Scale Logic」**：同时支持**对账灰度**与**切流灰度**；多维度如 **market、url、user、order、station**；支持白名单；精度可达 **万分之一（1/10000）**。
- **两阶段**：第一阶段部分场景（如 Opt 子域）切 LMO；第二阶段各子域与打 AT/RAT 等全面切 LMO。

### 6.5 指标监控

| 类型           | 指标示例                                                                |
| -------------- | ----------------------------------------------------------------------- |
| **业务** | LMO 查询 QPS、事件发送失败率、读对比不一致数、数据验证/状态扭转异常数。 |
| **服务** | DB QPS、主从延迟、Kafka 堆积、服务 CPU/内存、接口 RT。                  |

监控与告警阈值（如 RT &gt; 100ms、失败率 &gt; 0.1%）见 Confluence 非功能性设计章节。

| **量化指标** | LMO 查询 RT &lt; 100ms、事件处理 RT &lt; 200ms、失败率 &lt; 0.1%；ID 大促 DB 读峰值约 2.5 万 QPS（两年后约 5 万）；汇报口径 **LMO 平均 RT 较 FO 降低约 50%** | 用具体数字体现「性能与稳定性」 |

### 8.3 难点（体现技术深度与问题解决能力）

| 难点 | 具体内容 | 述职时怎么说 |
|------|----------|--------------|
| **1. 数据一致性** | 域内冗余数据必然存在不一致；需支持 FO/Plan 与域内异步任务**双源更新**，并处理乱序与幂等 | 「在域内冗余 FO 数据的前提下，设计并实现双源更新与乱序处理，保证最终一致性」 |
| **2. 乱序与幂等** | 多 Plan 类型（Delivery、Collection Pool、Return Locker、DataFix、Interception 等）的差异化逻辑；需「同 order 同 partition」「最后消费时间过滤」等保证顺序与幂等 | 「通过 Kafka 分区策略与消费时间过滤，保证多 Plan 类型下的乱序与幂等处理」 |
| **3. 容量与性能** | 十亿级单量、万级 QPS；需分库分表（10 库×100 表）、独立 DB、按市场 Kafka Topic；读 QPS = 接口×8 表、写 QPS = 接口×4 表 | 「按 shipment_id 做 10 库×100 表分库分表，支撑十亿级单量与万级 DB QPS」 |
| **4. 灰度与安全切流** | 读对比需连续 2 天 100% 一致后再切流；按 shipment_id 万分之一精度灰度；支持多维度匹配与白名单应急 | 「设计读对比 + 万分之一灰度方案，保证切流过程数据一致性」 |
| **5. 数据对账与修复** | 定时对账非终态 LMO 与 FO/作业信息；Plan 一致性校验；修数工具支持按 order id 批量创建/更新 LMO | 「设计数据对账与修数工具，支撑灰度与线上数据修复」 |

### 8.4 述职口述示例（30 秒～1 分钟）

> LMO 项目解决的是 FO 单点压力大、大促风险集中等问题。FO 承载十亿级订单数据，大促时部分场景到 FO DB 约 6 万 QPS，单点故障会影响全链路。  
> 我作为 PIC 负责 Delivery 域 LMO 从 0 到 1 的领域建模与生命周期设计，基于 Plan + Kafka 实现 FO/域内双源更新与乱序处理，并做了分库分表（10 库×百表 / 千表级方案以评审为准）、独立 DB 与按市场 Kafka Topic。  
> 难点包括：双源更新下的数据一致性、多 Plan 类型的乱序与幂等、以及读对比 + 万分之一灰度的安全切流。  
> 最终实现域内优先查 LMO、兜底 FO，降低对 FO 依赖；汇报口径 **LMO 平均 RT 较 FO 降低约 50%**，并支撑大促 DB 读峰值万级 QPS，LMO 查询 RT &lt; 100ms、失败率 &lt; 0.1%。

### 8.5 可追问的深挖点（提前准备）

1. **乱序如何保证幂等？** 同 order 同 partition、最后消费时间过滤等策略。
2. **双源更新如何保证一致性？** Plan 机制 + 域内异步任务，最终一致性保证。
3. **分库分表为何选 10 库×100 表？** 基于 FO 分库分表与峰值 QPS，如 ID 当前 DB 读约 2.5 万、两年后约 5 万。
4. **灰度为何用万分之一？** 控制影响面，支持多维度匹配与白名单应急。
5. **读对比为何要求连续 2 天 100% 一致？** 保证切流前数据与 FO 一致，降低风险。

## 九、简历/述职用 STAR 表述（示例）

*（与文首「金字塔总结」一一对应：背景 = 问题，方案 = 三大支撑之方案，成果 = 结果。）*

### 精简版（适合简历一栏）

- **S&T（问题）**：FO 承载**十亿级订单数据**（**ID 日常约 1000 万单、大促约 2000 万单**；**全市场日常约 2500 万单、大促约 5000 万单**），各域直连查 FO 导致**调用集中**（大促读 FO 接口峰值数千，如 ID 约 2000、VN 约 1200，部分场景到 FO DB 约 **6 万 QPS**）；单点故障影响全链路，需在域内落订单元数据以解耦并提升稳定性与性能。
- **A（方案）**：
  1. 负责 **Delivery 域 LMO** 的领域建模与生命周期设计。
  2. 基于 **Plan + Kafka** 实现 FO/域内双源更新与乱序处理；设计 **shipment_id 分片、10 库×100 表**（8 张表）、独立 DB 及按市场 Kafka Topic。
  3. 实现 **读对比 + 万分之一灰度** 的切流方案与数据对账、修数工具。
- **R（结果）**：域内使用 **LMO 替代 FO 的查询**（优先 LMO、兜底 FO），降低对 FO 依赖与 FO DB 压力；支撑**全市场日常约 2500 万/大促约 5000 万单、ID 日常约 1000 万/大促约 2000 万单**及 ID 大促 DB 读峰值约 **2.5 万 QPS**（材料另有 **50k/9k** 读写峰值口径，以封底与线上为准）；**LMO 平均 RT 较 FO 降低约 50%**（汇报口径）；LMO 查询 RT &lt; 100ms、事件 RT &lt; 200ms、失败率 &lt; 0.1%；具备完整灰度与数据一致性验证能力。

### 详细版（适合述职/面试展开，可口述关键数）

- **S&T**：同上；补充——单量维度 ID 日常 1000 万/大促 2000 万、全市场日常 2500 万/大促 5000 万；BR 批量接口到 FO DB 约 60K QPS，全链路单点压力大，大促风险集中。
- **A**：同上；补充封底与资源数据：
  - **容量与 QPS**：各市场半年单量预估（如 ID 4.92 亿条/470GB，两年后约 9.85 亿/939GB）；读 QPS = 接口×8 表、写 QPS = 接口×4 表；CAT 峰值×1.2 再预留 30% buffer。
  - **Kafka**：按市场 8 个 Topic，高流量市场单 Topic 约 2.4 MB/s、保留 2 天约 406 GB，核心链路不可延迟。
- **R**：同上；补充——ID 当前 DB 读约 **2.5 万**、写约 **4.7k**，两年后约 **5 万读 / 9.4k 写**；读对比 + 灰度验证保障数据一致性。
