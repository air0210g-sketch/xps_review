# LM Auto Assignment（末公里自动分单）

> 基于 SPX E2E Smart Solution / KP 项目材料整理，用于内部复盘与简历提炼。

---

## 1. 金字塔结构

### 1.1 塔尖（唯一结论）

在 **千万级日单、多 hub 并发触发** 的前提下，用 **Order Pool（Flink + ClickHouse）+ 版本化一致读 + 业务侧削峰与单站串行** 支撑自动分单规则，实现 **专送司机生产力与 CPO** 的可度量改善。

### 1.2 中层（三根支柱）

| 支柱 | 要解决的问题 | 核心手段 |
|------|----------------|----------|
| **实时性与可算** | 按 zone 聚合「在仓待分」订单量，满足规则引擎输入 | Order Pool：多源 Binlog → Flink → CK；读写分离、结果表预聚合 |
| **稳定性** | 2500+ hub 同时触发、大数据组件与业务库高压 | 批量写、分钟级预汇总、CK 部署与分区策略、单站单任务 + Kafka 控并发 |
| **数据正确性** | 多库多表、Binlog 延迟/乱序/丢失 | 单表 Binlog 版本、目标表版本 = max(多表版本)、异步合并 + 一致读，保证最终一致 |

### 1.3 底层（事实与结果）

- **业务结果**：专送生产力 **136 → 142（+4%）**；CPO **约 -2%**，约合 **400 万美元/年** 节约；rollout 站点 **约 40.7%**、对应 ADO **约 70%**。
- **系统形态**：新增 **Auto Assignment** 模块（入参 → 规则计算 → 产出/复用 AT）；扩展 **AT** 模块（建单/指派场景、zone 单量校验等）。
- **约束背景**：设计开发窗口紧（约 **4 周**）；关键需求仍在讨论，范围可能变化；大促等场景下单量与状态变更频率更高。

## 2. 重难点分析

### 2.1 业务与产品难点

- **目标矛盾**：要在「专送固定成本」与「产能打满」之间做系统化约束，把线下经验变成 **可配置规则 + 可执行** 的分配策略（如 zone 内 min/max 产能、优先级、专送/Mitra 分层）。
- **范围风险**：自动分单与 **订单域、配送域、运营流程** 耦合深，需求若在迭代中摇摆，会直接影响 Order Pool 口径与 AT 生命周期设计。

### 2.2 技术难点：Order Pool

- **规模**：日订单量级 **千万**；状态高频变更（如材料中所述 **A → B → A** 类回退）；还需按 **Station / Zone / 顺逆向 / 是否加急、大件、拦截** 等条件过滤再聚合。
- **时效期望**：本地用户对 **分区订单数** 的可见延迟敏感；需在 **秒级～分钟级** 与 **成本、一致性** 之间做显式取舍（材料中对比了 Flink+CK、Flink+Redis、Order Plan+MySQL、Listen Binlog+MySQL 等方案）。
- **落地选择（材料结论）**：采用 **Flink 监听多库 Binlog + ClickHouse**。利用 **订单明细表 + zone 计数结果表** 分离读写；Flink 侧批量写入、配送任务侧按分钟聚合写入结果表，读取走结果表以减压。

### 2.3 技术难点：稳定性与大数据组件

- **并发洪峰**：**2500+ hub** 可能同时触发自动分单，放大对业务接口、第三方依赖与 CK 的瞬时压力。
- **写入形态**：大量订单状态变更若 **逐条刷写** CK，易造成高频写入与 parts 过多；需 **合并多库操作、批量落库、预计算**（材料示例：等待 **30 秒或约 35w 条** 批量写入等思路）。
- **CK 运维**：独立部署、分区策略（材料提及由按天分区调整为 **5 天** 粒度等）以降低 parts 过多风险。

### 2.4 技术难点：一致性（多源 Binlog）

- **问题定义**：Order Pool 与 **3 个 DB、6 张表** 等源头的数据需在 **延迟、乱序、甚至丢失** 条件下仍可 **最终对齐**。
- **思路概要（材料）**：
  - **单表 Binlog 版本**：以 Binlog 事件时间戳等为版本，新事件覆盖旧状态，迟到小包可丢弃。
  - **目标版本**：`max(多表版本)`，版本推进后再 **Join 多表** 写入 CK。
  - **一致读**：异步合并多版本，读侧可拿到 **最新可见版本**。
- **收益**：在不大改业务事务的前提下，用 **Flink + CK 能力** 控制开发量，仍能保证 **最终一致**。

### 2.5 工程与协同难点

- **时间短**：设计开发约 **4 周**，需求仍可能变动。
- **应对（材料）**：**分期联调/测试**、产品侧引导 **更易落地的范围**、质量上 **压缩重复测试、强化用例/工具辅助**（如材料提到用 AI 工具生成单测、BE 侧重冒烟等）。

## 3. STAT 原则简历

以下为 **STAT** 框架版本：**S**ituation & **T**ask 合并叙述、**A**ction 分条、**R**esult 量化；数字与方案表述与答辩材料保持一致，对外投递前建议与业务/财务确认 **节约金额与统计口径**。

### S & T（Situation & Task）

末公里 **专送司机固定成本高**，线下分单难以稳定拉满专送产能，导致 **LM 成本不优**。业务需要一套 **自动分单（Auto Assignment）** 能力：在可配置的 **Hub/司机/Zone 规则** 下，把包裹分配给合适的司机，核心指标是 **提升专送司机生产力并降低 CPO**。同时，Order Pool 需在 **日单千万级、状态高频变更、多 hub 并发** 的条件下输出 **低延迟、可解释的分区单量**，并处理 **多源 Binlog 的延迟与乱序**。项目在 **极短周期（约 4 周）** 内交付，需求仍在讨论，必须用 **分期与范围管理** 控制风险。

### A（Action）

- **产品与范围**：与 PM/各域对齐 **分期上线** 策略，在需求未完全冻结时优先保证 **主路径可测、可灰度**，引导选择 **实现成本更低、边界更清晰** 的方案。
- **架构与模块**：主导新增 **Auto Assignment 模块**（输入参数、规则计算、输出并创建/复用 **AT**）；扩展 **AT 模块**（自动分单场景下的建单与指派、**zone 单量校验** 等）。
- **Order Pool 技术方案**：主导技术选型，落地 **Flink 消费多源 Binlog + ClickHouse**：
  - **读写分离**：订单/事件明细写入 CK 源表；**zone 订单计数** 写入独立结果表，读侧以结果表为主。
  - **稳定性**：通过 **批量写入、分钟级预聚合、CK 独立部署与分区策略** 等，缓解高频写入与高并发查询压力。
- **一致性**：设计 **Binlog 版本号 + 多表 Join + 目标版本推进 + 异步合并与一致读**，在 **延迟/乱序/丢失** 下仍保证 Order Pool **最终一致**。
- **业务洪峰治理**：推动 **每站单任务串行**、**Kafka 消费侧控并发**、失败场景的 **降级 / 重推 / 再消费** 等机制，避免同时 **2500+ hub** 打穿依赖。
- **质量与效率**：在工期紧的前提下，用 **工具辅助单测**、与 QA **减少重复劳动**，研发侧以 **冒烟 + 关键路径** 为主保障迭代速度。

### R（Result）

- **生产力**：专送司机生产力由 **136 提升至 142（约 +4%）**。
- **成本**：CPO **约降低 2%**，约合 **400 万美元/年** 成本节约（口径需业务确认）。
- **覆盖面**：功能 rollout **约 40.7%** 站点，对应 ADO **约 70%**。
