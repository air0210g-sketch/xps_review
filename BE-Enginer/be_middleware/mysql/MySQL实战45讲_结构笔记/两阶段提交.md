# 两阶段提交

**定义**：在一条更新语句的执行中，redo log 的写入被拆成两步——先写 redo log 并置为 **prepare**，再写 binlog，最后把 redo log 置为 **commit**。这样保证 redo 与 binlog 在「是否包含该事务」上逻辑一致。

**机制**：
- **顺序**：引擎写 redo（prepare）→ 执行器写 binlog → 引擎把 redo 改为 commit。
- **为何必要**：若先写 redo 再写 binlog，崩溃恢复后数据有该更新，但 binlog 未记，用 binlog 恢复/主备会丢该更新；若先写 binlog 再写 redo，崩溃后 redo 未记则本库恢复后无该更新，但 binlog 有，用 binlog 恢复会多出该更新。两阶段提交使「本库恢复结果」与「用 binlog 恢复/同步的结果」一致。
- **恢复时**：若 redo 为 prepare 且 binlog 完整，则提交该事务；若 redo 为 prepare 且 binlog 无，则回滚。

**语境**：理解「为什么 redo 要分 prepare/commit」「为什么能恢复到任意秒」以及主备一致、扩容用全量+binlog 时，都依赖两阶段提交。
