# 死锁

**定义**：多个事务循环等待对方持有的锁，都无法继续，形成死锁。

**机制**：
- **成因**：例如事务 A 持有 id=1 行锁并等待 id=2，事务 B 持有 id=2 并等待 id=1，则死锁。
- **策略**：① 超时等待：`innodb_lock_wait_timeout`（默认 50s）后回滚；② 主动检测：`innodb_deadlock_detect=on` 时发现死锁后回滚链条中某一事务，让其他继续。线上通常用主动检测。
- **代价**：高并发下对同一行更新时，每个被阻塞的事务都要做一次 O(n) 的死锁检测，CPU 飙升、TPS 下降。
- **缓解**：关死锁检测（有业务风险）；在服务端/中间件对「同一行更新」做排队限流；或把一行拆成多行（如影院账户拆成多条记录随机选一条更新）降低冲突。

**语境**：「只改一行」却锁很多、热点行更新导致 CPU 高而 TPS 低时，结合 [[行锁]]、[[间隙锁]] 看加锁范围，再考虑并发控制或拆行设计。
