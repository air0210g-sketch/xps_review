# 核心难点解剖：Redis 单线程模型

> 按 teach-tech 阶段二：第一性原理 & 3W1H。下钻到计算机科学原点（内存、IO、并发模型）。

---

## What（第一性原理：是什么？）

- **表象**：Redis 6.0 前，处理命令的「主路径」只有一个线程；6.0 起多线程只用于**网络 IO**（读写 socket），命令执行仍是**单线程**。
- **本质**：  
  - **命令执行**在单线程中顺序执行，无锁、无上下文切换，**所有数据结构操作在进程内内存完成**，本质是「单线程 + 内存访问 + 非阻塞 IO」的组合。  
  - 计算机原点：**CPU 执行指令**是纳秒级；**内存访问**是百纳秒级；**磁盘/网络 IO**是微秒～毫秒级。单线程把「等网络/等磁盘」从 CPU 上剥离出去，CPU 只做「算 + 内存读写」，因此单核就能打满。

---

## Why（为什么这么设计？）

1. **避免锁与竞争**：数据结构（dict、跳表、SDS）在单线程内修改，不需要加锁，无锁无竞争，代码路径简单、易维护。  
2. **避免上下文切换**：单线程无多线程切换成本，CPU 缓存局部性好，利于吞吐。  
3. **瓶颈不在 CPU**：多数命令是 O(1)～O(log N) 的内存操作，真正耗时的是网络 RTT 和持久化 IO；用多线程去「并行执行命令」收益有限，反而引入锁与复杂度。  
4. **历史与简洁**：早期设计选择单线程，后续在「保持单线程执行」前提下，只把网络 IO 多线程化（6.0），兼顾兼容与性能。

---

## Where（适用与不适用场景）

| 适用 | 不适用 |
|------|--------|
| 高 QPS、小 value、纯内存读写（缓存、会话、计数） | 单 key  value 巨大（大 key），单命令耗时长，会阻塞其他请求 |
| 对延迟敏感、需要可预测的 P99 延迟 | 需要多核并行做大量 CPU 计算（如复杂聚合） |
| 主从/集群下，单实例负责一部分槽，压力可水平拆分 | 单实例成为唯一热点且无法再分片 |

**边界**：单线程意味着「一个慢命令会拖垮整个实例」—— 避免 KEYS、大 key 的 HGETALL、长时间 Lua，用 SCAN、HSCAN、 pipeline 拆分。

---

## How（具体运行机制）

1. **事件循环**：主线程跑一个**事件循环**（基于 epoll/kqueue 等），只做两件事：  
   - 等 IO 就绪（客户端发来的 socket 可读、可写）；  
   - 就绪后读请求 → 解析 → **执行命令** → 写回响应。  
2. **IO 多路复用**：多个客户端连接复用一个线程，通过 **select/epoll** 在「可读/可写」时被唤醒，避免为每个连接开线程；本质是「等 IO 时不占 CPU」。  
3. **顺序执行**：命令在单线程内**严格顺序**执行，所以不需要对 key 加锁，也不会出现并发写同一 key 的竞态。  
4. **6.0 多线程**：多个 IO 线程负责 **read / write** socket；命令解析与执行仍在主线程，因此「命令原子性、单线程执行」语义不变，只是网络吞吐提升。

---

## 费曼检验（用人话复述）

- **一句话**：Redis 用「一个线程 + 多路复用」处理很多连接，命令在一个线程里排队执行，所以不用加锁；慢的是网络和磁盘，不是 CPU，所以单线程就能跑得很快。  
- **比喻**：一个前台（单线程）按顺序接待客人（命令），但客人把需求写在纸条上递进来（IO 多路复用），前台不用站着干等，可以一直处理手头的单子。
