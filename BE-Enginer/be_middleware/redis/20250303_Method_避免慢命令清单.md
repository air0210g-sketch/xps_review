# Redis 避免慢命令清单

## 🆔 Identifier
ID: 20250303_避免慢命令
Status: DRAFT

## 🎯 Purpose & Value

*   **Pain point**：单线程下一条慢命令会阻塞整实例，导致所有请求排队、延迟飙升甚至超时。
*   **Use case**：代码评审、上线前检查、故障排查时排除「大 key / 全表扫描 / 长 Lua」。
*   **Expected output**：禁止/替代命令清单 + 慢日志与 bigkey 检查步骤。

## ⚙️ Prerequisites

*   [ ] 可执行 redis-cli 或通过运维平台执行 CONFIG / SLOWLOG 等只读命令。
*   [ ] 已知业务使用的 key 命名与数据结构（便于识别大 key 与危险命令）。

## 📝 Actionable Steps

### Step 1：禁止与替代命令

| Step | What | How | Why |
|------|------|-----|-----|
| 1 | 禁止生产使用 KEYS | 用 SCAN 替代；若必须按模式查，用 SCAN 0 MATCH pattern COUNT 100 循环直到游标回 0 | KEYS 全库扫描，key 多时阻塞可达秒级 |
| 2 | 避免全量 HGETALL/SMEMBERS/LRANGE | 大 Hash 用 HSCAN；大 Set 用 SSCAN；大 List 用 LRANGE 指定区间或拆 key | 单 key 过大时一次返回阻塞主线程 |
| 3 | 控制 Lua 执行时间 | 脚本内避免大循环、避免 KEYS、用 SCAN；设置 lua-time-limit（默认 5000ms），超时仅中断当前脚本 | 长 Lua 占满单线程 |

### Step 2：慢日志与阈值

| Step | What | How | Why |
|------|------|-----|-----|
| 1 | 设定慢日志阈值 | CONFIG SET slowlog-log-slower-than 10000（10ms，单位微秒）；CONFIG SET slowlog-max-len 128 | 超过阈值的命令记入 SLOWLOG，便于事后排查 |
| 2 | 定期查看慢日志 | 执行 SLOWLOG GET 10，查看最近 10 条慢命令及耗时 | 发现 KEYS、大 key、异常聚合 |

### Step 3：大 key 识别与拆分

| Step | What | How | Why |
|------|------|-----|-----|
| 1 | 识别大 key | 用 redis-cli --bigkeys 或 RDB 分析工具扫描；或 MEMORY USAGE key 抽样 | 大 key 的读写/删除都会拉高延迟 |
| 2 | 拆分或迁移 | 将大 Hash/List/Set 按业务拆成多个 key（如 user:123:field1）；或异步迁移到新结构 | 单 key 体积控制在 KB 级为宜 |

## ✅ Acceptance Checklist

*   [ ] 代码与脚本中无 KEYS、无未限制范围的 HGETALL/SMEMBERS/LRANGE。
*   [ ] 慢日志阈值已设（如 10ms），且定期查看 SLOWLOG。
*   [ ] 已知大 key 列表，并已拆分或列入迁移计划。

## 🧠 Mechanism & Leverage

*   **Core mechanism**：单线程顺序执行，任一命令耗时 T 则该实例所有请求至少等待 T；避免 O(N) 全量操作、控制单 key 体积与 Lua 复杂度即可消除大部分慢命令。
*   **80/20**：KEYS + 大 key 全量读/写 + 长 Lua 覆盖绝大多数生产慢命令。

## 🧪 Next Step Experiment (MVE)

*   **Goal**：确认当前实例是否存在慢命令与大 key。
*   **Action**：在测试/预发执行 SLOWLOG GET 20；执行 redis-cli --bigkeys（读从库或低峰）；对返回的 command 与 key 做代码 grep。
*   **Expected result**：无 KEYS；无单 key 体积 >100KB 的 HGETALL/SMEMBERS/LRANGE；若有则列改进项。
