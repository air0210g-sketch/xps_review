## 1. 是什么？—— 第一性原理

### 1.1 一句话定义

**Redis** 是在**内存**中提供 **KV 存储 + 多种数据结构**的组件，解决「读写要快、能缓存、能当数据库/队列/锁」的矛盾；角色上既是**缓存**，也可做**持久化存储、消息队列、分布式协调**的底座。

### 1.2 整体架构

```
                    ┌─────────────────────────────────────────┐
                    │              客户端 / 应用               │
                    └─────────────────┬───────────────────────┘
                                      │
         ┌────────────────────────────┼────────────────────────────┐
         │ 单机                        │ 高可用/扩展                  │
         ▼                            ▼                            │
┌─────────────────┐    ┌──────────────────────────────────────────┤
│ 主线程           │    │ 主从复制 → 哨兵（故障转移）               │
│ · 事件循环       │    │ 或 集群（分片 + 高可用）                   │
│ · 命令执行       │    └──────────────────────────────────────────┘
│ · 单线程无锁     │
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
 内存 KV   持久化（RDB/AOF，子进程）
 数据结构   （不阻塞主线程）
```

- **单机**：主线程 = 事件循环（IO 多路复用）+ 顺序执行命令；持久化由子进程做 RDB 或 AOF 落盘。
- **高可用/扩展**：主从复制做数据同步 → 哨兵做主故障转移；或直接用集群做分片 + 每片主从。

### 1.3 数据流向图：从 Client 到 Server 磁盘

**写路径（Client → Redis 内存 → 磁盘）**：

```
Client 应用
    │
    ▼ 发送命令（如 SET key value）
┌───────────────────────────────────────────────────────────────────┐
│ 网络                                                               │
│   Client 内核：TCP 发送缓冲区 → 网卡 → 网络                          │
│   Server 内核：网卡 → TCP 接收缓冲区（套接字 buffer）                 │
└───────────────────────────────────────────────────────────────────┘
    │
    ▼ Redis 进程
┌───────────────────────────────────────────────────────────────────┐
│ 用户态                                                             │
│   主线程：epoll 读 socket → 解析命令 → 执行（写内存 dict 等）         │
│   数据落点：内存 KV 数据结构（如 dict、跳表）                         │
└───────────────────────────────────────────────────────────────────┘
    │
    ├─────────────────── AOF 路径 ───────────────────┐
    │  主线程：命令追加到 AOF buffer（内存）              │
    │    → 后台/定时由主线程或 bio 线程 write() 到 OS    │
    │    → 内核：Page Cache（文件系统缓存）              │
    │    → appendfsync 策略：everysec/always 触发 fsync  │
    │    → 磁盘                                          │
    │                                                    ▼
    └─────────────────── RDB 路径 ──────────────────── 磁盘
       fork 子进程：读主进程内存（COW）→ 序列化
         → write() 到临时 RDB 文件
         → 内核：Page Cache → fsync 落盘 → rename 为正式 dump.rdb
```

**读路径（Client 读 Redis）**：

```
Client 请求（如 GET key）
    → 网络（同上）
    → Server 内核 TCP 接收缓冲区
    → 主线程 epoll 读 socket → 解析 → 在内存数据结构中查找
    → 结果写入 TCP 发送缓冲区 → 经网络返回 Client
（读路径不经过磁盘，除非从库加载 RDB/AOF 做启动恢复）
```

**要点**：写命令先落**内存**，再通过 AOF buffer / 子进程 RDB 经 **write → 内核 Page Cache → fsync** 落盘；持久化不阻塞主线程命令执行。

## 4. 出现故障，如何排查？

| 故障类型 | 排查方向 | 工具/命令/笔记 |
|----------|----------|----------------|
| **慢查询 / 阻塞** | 单线程被一条命令占满 | SLOWLOG、避免 KEYS/大 key 全量、Lua 超时 | [[20250303_Method_避免慢命令清单]] |
| **内存爆 / 满** | maxmemory、淘汰策略、大 key、碎片 | INFO memory、redis-cli --bigkeys、MEMORY USAGE |
| **主从延迟 / 断线** | 复制偏移、repl_backlog 被覆盖、网络 | INFO replication、repl_backlog_size |
| **主从/哨兵脑裂** | 分区后多主、min-replicas-to-write | 哨兵配置、客户端多写检测 |
| **集群槽迁移/路由异常** | MOVED/ASK、槽未完全迁移 | CLUSTER SLOTS、CLUSTER NODES、hash tag |

可收敛为一份 **《Redis 故障排查清单》**（SOP），在后续 Method Note 中补全。
