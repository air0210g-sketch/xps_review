# Redis 分布式锁实现检查清单

## 🆔 Identifier
ID: 20250303_分布式锁实现
Status: DRAFT

## 🎯 Purpose & Value

*   **Pain point**：实现不当会导致误删他人锁、死锁、或锁过期后多客户端同时持有，失去互斥语义。
*   **Use case**：实现或评审「基于 Redis 的分布式锁」时逐项核对。
*   **Expected output**：加锁/续期/释放的合规实现 + 检查项勾选。

## ⚙️ Prerequisites

*   [ ] 已确定锁的 key 命名与业务超时上界（用于设 PX/EX）。
*   [ ] 客户端能执行 SET/GET/Lua 或使用 Redisson 等已实现锁的库。

## 📝 Actionable Steps

### Step 1：加锁

| Step | What | How | Why |
|------|------|-----|-----|
| 1 | 使用 SET 且带 NX + 过期 | SET lock_key unique_value NX PX 30000（30 秒过期）；unique_value 为 UUID 或唯一 ID | NX 保证仅不存在时写入；PX 避免宕机死锁 |
| 2 | 将 unique_value 与锁绑定 | 加锁成功后本地保存 unique_value，释放时必须用该值校验 | 防止释放时误删其他客户端的锁 |

### Step 2：释放锁（原子校验再删）

| Step | What | How | Why |
|------|------|-----|-----|
| 1 | 只删自己持有的锁 | 用 Lua：if redis.call("get", KEYS[1]) == ARGV[1] then return redis.call("del", KEYS[1]) else return 0 end；KEYS[1]=lock_key，ARGV[1]=unique_value | GET 与 DEL 必须在同一次执行中，否则并发下会删掉他人锁 |
| 2 | 避免仅 DEL 不校验 | 禁止在未校验 value 的情况下直接 DEL lock_key | 若锁已过期被他人获取，直接 DEL 会删掉新锁 |

### Step 3：续期（长任务可选）

| Step | What | How | Why |
|------|------|-----|-----|
| 1 | 在持有期内定时续期 | 每 (ttl/3) 毫秒执行 PEXPIRE lock_key ttl，直到业务完成或主动释放 | 业务执行时间可能超过初始 ttl，续期避免中途锁过期 |
| 2 | 仅对「自己持有的锁」续期 | 续期前用 GET 校验 value 是否仍为自己的 unique_value；或使用 Redisson 的 watchDog | 避免为他人锁续期 |

### Step 4：单点与多实例（可选）

| Step | What | How | Why |
|------|------|-----|-----|
| 1 | 单 Redis 时接受单点风险 | 主从异步复制下，主宕机可能导致从尚未复制到锁就被提升，出现双持；可接受则用单实例即可 | 多数业务可接受极短时间窗口的双持风险 |
| 2 | 需更强保证时评估 Redlock | 在 N 个独立实例上多数派加锁（N/2+1）；需评估时钟与网络假设 | 降低单点影响，但存在争议与实现成本 |

## ✅ Acceptance Checklist

*   [ ] 加锁使用 SET key value NX PX/EX ttl，且 value 唯一。
*   [ ] 释放使用 Lua 或等价原子逻辑，先校验 value 再 DEL。
*   [ ] 若业务执行时间可能超过 ttl，已实现续期且仅续期自己的锁。
*   [ ] 已明确单实例还是多实例（Redlock），并知晓对应风险。

## 🧠 Mechanism & Leverage

*   **Core mechanism**：互斥靠 NX 的「仅一次写入」；安全释放靠「唯一 value + 原子校验删除」；可用性靠过期时间与续期。
*   **80/20**：多数问题出在「未设过期」或「释放时未校验 value」，先做好这两点再考虑 Redlock。

## 🧪 Next Step Experiment (MVE)

*   **Goal**：验证同一 key 只能被一个客户端持有，且释放后他人可再次获取。
*   **Action**：两进程/线程同时加锁同一 key，确认仅一个成功；成功方释放后，另一方再次加锁应成功；成功方用错误 value 执行释放 Lua，锁应仍存在。
*   **Expected result**：互斥成立；错误 value 无法删锁；释放后锁可被再次获取。
