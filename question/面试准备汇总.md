## 二、自我介绍

### 2.1 一分钟版（推荐口播 · 已按简历定制）

您好，我是周佳灿，9 年研发经验，求职后端方向。近四年在**虾皮物流 SPX** 做**国际末端派送核心域**，生产环境以 **Golang** 为主，也有多年 **Java** 背景。

在 SPX 我主要做了三件事：一是 **LMO（Last Mile Order）**，在 FO 承载千万～大促全市场约五千万级日单、各域直连导致单点压力的背景下，负责派送域内**订单元数据**的建模、生命周期和分库分表，用 **Plan + Kafka 双源更新** 处理乱序幂等，配合**多阶段对账与万分之一起步的灰度**，把域内查询迁到 LMO，**平均 RT 较原方案约缩短一半**，并支撑大促峰值读写。二是 **印尼自动分单**，主导 **Auto Assignment** 与 **AT** 扩展，用 **Flink 多源 Binlog + ClickHouse** 做 **Order Pool**，在千万级日单和多 hub 并发下保证可解释、低延迟与最终一致，业务上专送生产力与成本有可量化结果。三是 **大单体按域拆分与 Python 迁 Go**，引入 **Plan 与域内状态机、DDD 聚合**，**读对比加接口灰度**平滑切流，核心接口 **RT 约降七成、机器约降至原先三分之一**。

我希望继续在**高并发、强工程治理的物流与交易履约链路**上深耕，把大流量下的**一致性与稳定性**经验带到贵团队。谢谢。

### 2.2 三十秒版（精简）

我是周佳灿，9 年经验，后端方向。近年在**虾皮 SPX 末端派送域**：负责 **LMO** 解耦 FO（双源更新、对账、万分之一灰度，查询 RT 约降一半）；主导 **印尼自动分单**（Flink + 多源 Binlog + ClickHouse Order Pool，生产力与成本可量化）；并推动 **Python 大单体按域拆微服务 + 迁 Go**（核心 API RT 约降七成、机器约原三分之一）。熟悉高并发下的最终一致与灰度治理。希望继续在物流履约与高可用后端方向深耕。

### 2.3 三分钟版（提纲）

在 **一分钟版** 基础上，每条经历各扩展 **一句难点 + 一句量化**：

1. **重构**：难点是 **强一致 → 最终一致** 与 **大量接口读对比灰度**；可提 **Cancel Plan 终结流程、FO 按 mtime 更新**、**Facade/HTTP 边界**。
2. **LMO**：难点是 **双源更新、乱序幂等、安全切流**；可提 **印尼事件写峰值约 4K QPS**、**ID 大促 DB 读峰值万级 QPS（封底/两年后更高，以线上为准）**。
3. **自动分单**：难点是 **多源 Binlog 延迟/乱序/丢失下的最终一致** 与 **CK 写入与查询压力**；可提 **批量写、分钟级预聚合、单站串行、Kafka 控并发**。

结尾补充 **应聘公司/业务匹配**（1～2 句），避免流水账。

## 三、STAR 故事库

### 故事 1：Python 大单体 → Go 微服务 + Plan 解耦 FO

| 模块 | 内容 |
|------|------|
| **S** | Python 共仓大单体 + 目录隔离，性能瓶颈、模块耦合、发布冲突；强依赖 **FO 状态机**，并发场景（如 onhold 下 redelivery + hub receive）易不一致，频繁 datafix。 |
| **T** | **不中断业务**前提下：服务拆分；「FO 强校验」→ **事件驱动域内校验 + 最终一致**；核心 **Python→Go**。 |
| **A** | 按作业域拆微服务，**Facade/HTTP** 边界；**Plan** + LM 域内状态机；**Kafka/轨迹** 驱动 FO 按 **mtime** 更新；**AT/TO 解耦**、**Cancel Plan** 终结；**DDD** 与 **DeliveryOrder** 聚合根；**Go 重写** + **读对比（≥99.99%）** + **接口灰度**（随机比例、可降级）。 |
| **R** | 核心 API **P99 明显下降**（如 Delivery 99 线 **12087→5742ms** 量级，Mass Receive **4 万→约 1.3 万 ms**）；机器约 **原 1/3**；单点不拖垮整体，并发 datafix 问题缓解。 |

**常见追问**：为何最终一致可接受？迁移如何不掉单？Cancel Plan 为何先终结 DO？

### 故事 2：LMO — 域内订单元数据

| 模块 | 内容 |
|------|------|
| **S** | FO 十亿级订单；全市场日常约 **2500 万单/日**、大促约 **5000 万**；直连 FO 致调用集中、大促 **FO DB 高 QPS**；FO 故障影响全链路。Delivery 缺覆盖全作业周期的域内模型。 |
| **T** | 引入 **LMO**，**优先 LMO、兜底 FO**；支撑 **十亿级数据 + 万级 DB QPS**；查询 **RT &lt;100ms**、失败率 **&lt;0.1%**（目标口径）。 |
| **A** | **PIC**：领域模型与生命周期；**Plan + Kafka** 双源；**同 order 同 partition、操作时间、状态机、限时重试**；**10 库×100 表（8 逻辑表）**、独立 DB、按市场 Topic；**读对比**（如连续 2 天一致或 **99.99%** 对账，以门禁为准）+ **万分之一灰度** + 对账/修数。 |
| **R** | 降 FO 依赖与 DB 压力；汇报口径 **LMO 平均 RT 较 FO 约降 50%**；简历亦写 **对账一致性达 99.9% 后灰度**（与材料中 99.99% 表述并存时，面试说明**以当时门禁为准**）。 |

**常见追问**：乱序如何保证幂等？灰度如何回滚？不一致如何修？

### 故事 3：LM Auto Assignment — 自动分单 + Order Pool

| 模块 | 内容 |
|------|------|
| **S** | 专送固定成本高；需 **可配置规则** 自动分单；**千万级日单**、状态高频变更、**2500+ hub** 同触；多源 Binlog **延迟/乱序/丢失**；工期约 **4 周**、需求可能变。 |
| **T** | 交付 **Auto Assignment**（入参→规则→**AT**）；扩展 **AT**（建单/指派、zone 单量校验）；建设 **Order Pool**（低延迟、可解释、最终一致）。 |
| **A** | 分期上线、锁主路径；**Flink + 多源 Binlog + CK**；明细表与 **zone 计数结果表** 分离；批量写、分钟级预聚合、分区策略；**Binlog 版本 + max 多表版本 + 异步合并一致读**；**单站串行、Kafka 控并发**、降级/重推/再消费；工具辅助单测与冒烟。 |
| **R** | 生产力 **136→142（约 +4%）**；CPO **约 -2%**，约合 **400 万美元/年**（**须业务/财务确认口径**）；rollout **约 40%**、简历写 **对应每日单量提升约 70%**（与材料中 ADO 表述对照时统一口径）。 |

**常见追问**：为何 Flink+CK 而非 Redis？大促如何扩容？规则变更对 Order Pool 的影响？

## 四、虾皮 / SPX 业务向面试题准备

### 4.1 业务场景与「为什么做」

| 可能问题 | 考察点 | 回答要点 |
|---------|--------|----------|
| FO、LMO、AT 各解决什么？与 Shopee 订单关系？ | 域划分 | FO：履约元数据中枢；LMO：Delivery **域内**查询，优先 LMO 兜底 FO；AT：派送任务，自动分单结果落 AT。 |
| 为何做自动分单？生产力与 CPO 如何理解？ | 指标与取舍 | 专送固定成本高，要产能与成本平衡；规则配置化；生产力、CPO 可运营复盘。 |
| 印尼与其它市场差异？ | 区域化 | 简历：**印尼事件写峰值约 4K QPS**；Topic/容量按市场拆；无一线细节则诚实说明。 |

### 4.2 指标与结果（简历数字易被追问）

| 可能问题 | 考察点 | 回答要点 |
|---------|--------|----------|
| RT 降 50% 与对账 99.9% 分别指什么？ | 口径 | RT：查询路径与 FO 压力；99.9%：当时切流门禁；可与材料 99.99% 区分说明阶段。 |
| 生产力、400 万刀、40% rollout、70% 单量提升如何定义？ | 因果与统计 | 定义以业务/数分为准；金额为年化估算；rollout 与全局单量**勿过度归因个人**。 |
| 成本节约是否全归你团队？ | 协作 | 系统 + 运营规则 + 覆盖范围共同作用；你侧强调可交付、可观测部分。 |

### 4.3 规则、产品与跨团队

| 可能问题 | 考察点 | 回答要点 |
|---------|--------|----------|
| 规则谁定？需求频繁变更怎么办？ | 协作与范围 | PM/运营；分期 MVP；Order Pool **口径变更成本高**，早锁或版本化。 |
| zone 单量慢或错的影响？ | SLA | 影响规则输入；延迟与一致性权衡；降级重推；监控与可解释输出。 |

### 4.4 稳定性与故障

| 可能问题 | 考察点 | 回答要点 |
|---------|--------|----------|
| 大促/多 hub 同触最怕打穿哪里？ | 洪峰 | CK/Flink/业务接口；批量写、预聚合、分区；单站串行、Kafka 控并发；降级重推。 |
| LMO 切流不一致预案？ | SOP | 灰度维度；读对比与回滚；修数与对账。 |
| FO 抖动时体验？ | 降级 | 兜底 FO；超时与监控。 |

### 4.5 行为与组织

| 可能问题 | 考察点 | 回答要点 |
|---------|--------|----------|
| 与本地运营/PM 不一致？ | 沟通 | 对齐指标与约束；用数据推动；分期方案。 |
| 紧周期如何排期砍需求？ | 优先级 | MVP 可灰度；非核心后置；关键路径冒烟与工具提效。 |

### 4.6 建议自备「数字 + 名词」

- **数字**：2500 万 / 5000 万（全市场日单量级）；印尼 **~4K QPS** 事件写；生产力 **136→142**；资源 **~1/3**；RT **~50% / ~70%**（分清 LMO 与重构）。  
- **名词**：Plan、双源更新、万分之一灰度、Order Pool、最终一致、AT。
