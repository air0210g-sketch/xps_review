### 项目经历

#### Shopee SPX 国际末端派送平台

**1）派送订单领域模型 LMO（Last Mile Order）**

- **背景**：FO 模块承载日常千万级订单量（大促峰值全市场可达约 5000 万单），各域直连 FO 导致调用集中，FO 抖动影响派送 RT，单点故障影响全链路。需在派送域内落地订单元数据以解除对 FO 的依赖；在约半年、约 **10 亿级**数据量背景下保障一致性与准确性，支撑子领域查询。  
- **职责**：  
  - 负责 Delivery 域 LMO 的领域建模、生命周期管理及分库分表设计。  
  - 印尼市场事件写入峰值约 **4K QPS**，采用 Plan + Kafka 实现域内外双异步源更新 LMO；通过同订单同 partition、按操作时间、状态机校验、限时重试完善乱序与幂等。  
  - 建设监控告警与多阶段数据对账，一致性到达 **99.9%** 后开始灰度切流。  
  - 万分之一起步的灰度方案完成平滑切换，支持单号、站点、用户、场景等多维度组合灰度。  
- **成果**：域内以 LMO 替代 FO 查询，平均查询 RT 较 FO **缩短约 50%**；解除对 FO 服务依赖并降低 FO DB 压力；支撑全市场日常约 **2500 万单**（大促约 **5000 万单**）及 ID 单市场大促 DB 读写峰值；**零故障**完成 LMO 平滑切换，具备完整灰度与数据一致性验证能力。  

**2）印尼自动分单（Auto Assignment）**

- **背景**：东南亚末端专送场景下需自动分单能力：在可配置的司机/Zone 规则下分配包裹，核心指标为提升专送司机生产力并降低每单派送成本。挑战在于日单千万级、状态高频变更、多站点并发下的低延迟与可解释输出，以及多源 Binlog 的延迟与乱序。  
- **职责**：  
  - 主导新增 Auto Assignment 模块（输入、规则计算、输出并创建/复用 AT）；扩展 AT 模块（自动分单建单与指派、zone 单量校验等）。  
  - 亿级数据量 Order Pool：Flink 监听多库 Binlog + ClickHouse；订单明细表与 zone 计数结果表读写分离；Flink 侧批量写入 CK，配送任务侧按分钟聚合写结果表，读侧走结果表减压。  
  - 一致性：以 Binlog 事件时间戳等为版本，新事件覆盖旧状态，迟到事件可丢弃，异步合并多版本写入 ClickHouse，多库多表源在延迟、乱序甚至丢失条件下仍可最终对齐。  
  - 稳定性：Flink 批量写入、ClickHouse 分钟级预聚合、分区策略与定时任务等缓解高频写入与高并发查询；业务洪峰下推动每站任务串行、Kafka 消费侧控并发、失败场景的降级/重推/再消费，避免超大规模 hub 打穿 ClickHouse。  
- **成果**：专送司机生产力由 **136 提升至 142**（约 **+4%**）；成本节约约合 **400 万美元/年**；功能 rollout 约 **40%** 站点，对应每日单量提升约 **70%**。  

**3）大单体服务架构升级及重构**

- **背景**：SPX 派送与站内等模块原为 Python 大单体，存在性能瓶颈、跨模块耦合与发布冲突；强依赖 FO 状态机导致并发场景不一致与频繁 datafix。目标：持续迭代下完成服务拆分、解耦（强一致→最终一致）与 Python→Go 迁移。  
- **职责**：  
  - 主导派送领域架构升级：大单体按作业域拆微服务，Facade/HTTP 边界，独立部署治理；领域解耦，落地 DDD 限界上下文与 DeliveryOrder 聚合根；引入 Plan 与调度中心，Plan 生命周期内由 LM 域 DO 状态机管理域内流程，域外通过轨迹事件驱动更新，保证数据最终一致。  
  - 派送工程以 Golang 重写核心服务；读对比（一致率 **≥99.99%**）+ 接口维度灰度（Apollo、随机比例、可降级）平滑切流并下线 Python。  
- **成果**：核心 API RT **下降约 70%**；机器资源约 **原 1/3**；域间依赖与代码冲突减少，单点故障不再导致整体不可用。
